先建立一个可用的心智模型

读 Godot 源码时，第一件事不是记住每个目录，而是先把“用户看到的对象”“运行时保存的状态”和“底层真正执行的系统”分开。 一个场景能跑起来，至少要同时看四条轴线：启动轴、对象轴、场景轴、服务轴。启动轴把平台进程变成一个运行中的 MainLoop； 对象轴让 C++ 类能被脚本、编辑器、Inspector 和资源系统识别；场景轴把资源实例化成 Node 树并安排每帧处理；服务轴把渲染、物理、音频、文本、显示等重活交给可替换的后端。 这里的“轴线”不是目录层级，而是读源码时反复追问的四类问题：谁启动进程，谁注册类型，谁在 SceneTree 里保存用户语义，谁在 Server 或平台后端真正执行。

用户可见世界 反射与数据契约 场景运行时 Server 与后端

这个模型怎么用

对初学者来说，先把五个词分清：Node 是场景树里的运行对象，Resource 是可加载、可保存的数据，Script 是挂到 Object/Node 上的行为，SceneTree 是每帧调度节点的 MainLoop，Server 是渲染、物理、音频、文本、窗口等底层服务的统一入口。读任何功能时，都可以问：用户看到的名字是什么？状态保存在 Node 还是 Resource？每帧由 SceneTree 的哪个阶段推进？最后交给哪个 Server 或平台后端执行？

中级阅读时要把依赖方向看清楚：scene 可以调用 servers，两者都依赖 core；core 不能反过来依赖 scene。modules 不是单独一层，它会按初始化级别插入 core、servers、scene、editor：例如 gdscript 接入脚本语言，jolt_physics 接入 3D 物理后端，gltf 接入导入导出，visual_shader 接入编辑器和 shader 资源。

读到底层实现时，重点不是“某个类在哪里”，而是“契约在哪里定义”：ClassDB 定义 C++ 类型怎样暴露给脚本和编辑器；Variant 定义跨语言值怎样传递；RID 定义场景层怎样引用 Server 内部对象；MessageQueue 定义延迟调用什么时候安全执行；Main::iteration 定义每帧阶段顺序；register/unregister 定义类型和模块的生命周期边界。

判断一段源码的职责位置时，不要只看文件夹名，还要看它持有什么状态、被谁调用、在哪个初始化级别注册、每帧在哪个阶段执行。例如 MeshInstance3D 在 scene，但它真正可见要靠 RenderingServer 中的 RID；GDScript 在 modules，但它通过 ScriptLanguage、ClassDB、Variant 接入 core 的对象世界。

把抽象落到一次真实动作：点击运行主场景

下面这条链路是读 Godot 源码最有用的“坐标系”。它从一个普通动作开始：用户在编辑器里点击运行主场景。 这个动作横跨两个阶段：编辑器进程先保存、构建、拼运行参数并启动游戏进程；游戏进程再从平台入口进入 Main，把主场景作为 PackedScene 加载并交给 SceneTree。 把这两段分开，才能解释为什么编辑器按钮、调试器、游戏主循环和主场景加载分别落在不同源码区域。

1
平台进程先进入 OS

可执行文件从 platform/windows/godot_windows.cpp:68 这类平台入口开始，创建 OS_Windows 等平台 OS 对象，再调用 Main::setup()。这个阶段解决编码、命令行、系统 API、崩溃处理和平台事件循环的差异。

2
Main 建立运行时地基

main/main.cpp:1027 的 Main::setup() 开始解析命令行和建立 core 层，核心类型注册在 main/main.cpp:1059，CORE 级模块初始化在 main/main.cpp:2255，主 MessageQueue 在 main/main.cpp:2899 创建。没有这一步，后面任何 Resource、Object、Variant、脚本语言和项目设置都没有统一入口。

3
Server 与 Scene 类型接入

main/main.cpp:3008 的 Main::setup2() 进入第二阶段，调用 register_server_types()、register_scene_types()，并按 CORE/SERVERS/SCENE/EDITOR 初始化模块。这里决定 RenderingServer、PhysicsServer、AudioServer、Node、Control、PackedScene、AnimationPlayer 等类型能否被创建和绑定。

4
编辑器本身也是场景树程序

如果以编辑器运行，EditorNode 是主控节点，源码锚点是 editor/editor_node.h:120。运行按钮、文件系统 Dock、Inspector、场景树 Dock、导入导出面板都在这棵编辑器场景树里。也就是说，编辑器不是 Godot 外面的工具，而是 Godot 运行时里的一套复杂工具场景。

5
运行按钮启动游戏进程

点击运行主场景时，编辑器侧入口在 editor/run/editor_run_bar.cpp:256。它确认主场景、自动保存、调用构建和插件扩展点，然后在 editor/run/editor_run_bar.cpp:353 进入 EditorRun::run()。真正创建游戏进程的是 editor/run/editor_run.cpp:182 的 OS::create_instance()；如果指定场景，会在 editor/run/editor_run.cpp:150 到 153 追加 --scene 参数。

6
游戏进程选择 MainLoop 和主场景

新进程会重新走平台入口、Main::setup()、Main::setup2() 和 Main::start()。在 main/main.cpp:4335 附近会创建或选择 MainLoop，游戏路径来自命令行 --scene 或 application/run/main_scene，主场景加载逻辑集中在 main/main.cpp:4735 到 4743。

7
主场景作为 Resource 被加载

项目主场景路径最终交给 ResourceLoader。它在 core/io/resource_loader.h:103 定义统一加载入口，通过 ResourceFormatLoader 识别 .tscn、.scn、.tres 等格式，返回 PackedScene 或其他 Resource。

8
PackedScene 实例化成 Node 树

PackedScene 保存的是可实例化的 SceneState，源码锚点是 scene/resources/packed_scene.h:246。实例化时会创建 Node，恢复属性、子节点、信号连接、脚本实例和资源引用。Node 的核心声明在 scene/main/node.h:54。

9
SceneTree 按阶段调度

SceneTree 是 MainLoop 的具体实现，源码锚点是 scene/main/scene_tree.h:89。它负责 enter_tree、ready、process、physics_process、组调用、定时器、Tween、场景切换、删除队列和消息队列 flush。用户脚本的 _ready()、_process()、_physics_process() 都落在这条调度线上。

10
每帧由 Main::iteration 推进

main/main.cpp:4896 的 Main::iteration() 计算 delta 和固定物理步，调用 SceneTree 的 physics/process，刷新 MessageQueue，推进 NavigationServer，再调用 RenderingServer::sync() 和 RenderingServer::draw()。这就是“游戏在跑”的真正时间轴。

11
重活交给 Server 和后端

渲染对象通过 RenderingServer 的 RID API 进入后端，源码锚点是 servers/rendering/rendering_server.h:64；物理对象通过 PhysicsServer2D/3D 进入 Godot Physics 或 Jolt；音频通过 AudioServer 混音；文本通过 TextServer shaping。场景层只保存用户语义和句柄，底层系统保存真实执行状态。

四个“不要混淆”

容易混淆实际区别
Object 和 NodeObject 是反射和信号的根；Node 是能进入 SceneTree 的 Object 子类。Resource 也是 Object 体系，但不能挂到树上。
Resource 和 RIDResource 是可保存、可引用计数、可被 ResourceLoader 管的用户资源；RID 是 Server 内部资源的轻量句柄，不负责序列化。
scene 和 serversscene 保存用户语义和节点关系；servers 保存底层执行对象。MeshInstance3D 不是 GPU mesh，RigidBody3D 也不是物理求解器本体。
modules 和 thirdpartymodules 是 Godot 的功能接入层；thirdparty 是外部库源码。真正读 Godot 行为时，先读模块封装，再看第三方库。

五个定位问题

  • 它怎么被创建？查注册函数、ClassDB、ResourceLoader、PackedScene 或 EditorPlugin。
  • 它的数据存在谁身上？看头文件成员变量，区分 Node 状态、Resource 状态、Server RID 状态。
  • 它什么时候执行？回到 Main::iteration、SceneTree::process、SceneTree::physics_process、Server::draw/step。
  • 它怎么暴露给脚本和 Inspector？找 _bind_methods()、PropertyInfo、MethodBind、ADD_PROPERTY、ADD_SIGNAL。
  • 它能否被裁剪或替换？看 SCons 选项、模块宏、初始化级别、ServerManager 或 GDExtension 接入点。

源码地图：每个顶层目录负责什么

下面的源码地图按“职责边界”来读，而不是按文件夹名字背目录。Godot 的目录不是平均切分的：core 是运行时地基， scene 是用户可见对象层，servers 是底层能力抽象，modules 是可裁剪功能接入点， platform 和 drivers 把抽象落到系统 API。读源码时要先判断“这个功能的入口在哪个职责位置，真实执行在哪个系统，注册在哪个阶段发生”。 这里的“入口”指用户 API、编辑器按钮、命令行或资源格式的最外层名字；“真实执行”指持有状态或运行算法的对象；“注册阶段”指 ClassDB、Server 或模块初始化让它变得可见的时机。

从一个功能反查源码的固定步骤

1
先确定用户 API 名字

从编辑器看到的类名、节点名、资源名或脚本方法开始，例如 MeshInstance3D、AnimationPlayer、ResourceLoader.load()、RigidBody3D。先用 rg "class MeshInstance3D" 或 rg "_bind_methods" scene/3d 找到用户 API 的 C++ 类。

2
看它怎样暴露给脚本和编辑器

进入类的 _bind_methods()，看 ClassDB::bind_method、ADD_PROPERTY、ADD_SIGNAL。这一步能回答“为什么 Inspector 有这个属性”“为什么 GDScript 能调用这个方法”。相关底层在 core/object/class_db.h 和 core/object/method_bind.h。

3
看头文件里的真实状态

头文件成员变量比函数名更能说明职责。Node 类通常保存树关系、属性、资源引用、RID 句柄或缓存；Resource 类保存可序列化数据；Server 类保存抽象 API；backend 类保存真正执行状态。先看 *.h，再看 *.cpp 的变更路径。

4
找到运行时推进点

如果是每帧行为，回到 Main::iteration()、SceneTree::process()、SceneTree::physics_process()、RenderingServer::draw() 或 PhysicsServer::step()。很多看似“立即执行”的 API 实际只是改状态，真正执行在下一次 flush/sync/draw/step。

5
沿着 Server 或 Resource 边界继续追

如果看到 RID、RenderingServer、PhysicsServer、AudioServer，说明场景层已经把工作交给底层服务；如果看到 Ref<Resource>、ResourceLoader、ResourceFormatLoader，说明要进入资源加载/保存/导入管线。

6
最后确认模块和平台条件

如果源码被 #ifdef MODULE_*、TOOLS_ENABLED、PHYSICS_3D_DISABLED、平台宏包住，必须回到 SConstruct、modules/SCsub、modules/modules_builders.py、platform/*/detect.py 确认它是否参与当前构建。register_module_types.gen.cpp 是构建时生成文件，本地源码树里不一定已经存在。

构建系统：SCons 如何把源码拼成引擎

Godot 不是 CMake 项目，而是 SCons 项目。读构建系统时不要把它想成“一个工程文件列出所有源码”，而要把它想成一条 Python 驱动的装配线： 根目录的 SConstruct 先判断当前机器能构建哪些平台、解析用户传入的参数，再把同一个 env 传给各层 SCsub。 每个目录的 SCsub 把自己的源文件、生成文件、第三方库和宏开关加入对应的 source 列表，最后平台层把这些对象链接成 editor 或 export template。 SConstruct/SCsub 是构建时运行的 Python 脚本，不是引擎运行时逻辑；它们决定哪些 C++ 文件会被编译、哪些宏会被定义、哪些生成文件会参与编译。 所以构建系统不是“源码之外的杂项”，它直接决定哪些类存在、哪些模块注册、哪些宏包住的代码会进入最终二进制。

构建系统的可用心智模型

一次 scons platform=windows target=editor 发生什么

1
SConstruct 先建立构建环境

SConstruct 被 SCons 当作 Python 脚本执行。它加载 methods.py，并在 SConstruct:128 把 methods.add_source_files 挂到 env 类上。之后每个 SCsub 调用的 env.add_source_files(...) 都会进入 methods.py:79，这里就是“目录把源码交给构建系统”的统一入口。

2
扫描可构建平台

SConstruct:70 建立 platform_list，随后遍历 platform/*。只有目录里有 detect.py，并且 detect.can_build() 通过的平台才会在 SConstruct:103 加入列表。你传入的 platform=windows 最终会在 SConstruct:414 被校验。

读构建错误的定位顺序

平台不在列表里

模块没有编译

确认模块目录是否有 config.py、SCsub、register_types.h，再查 module_<name>_enabled、modules_enabled_by_default 和 env.module_list。

启动链路：从 exe 到每一帧

Godot 的启动代码很长，但不是一团乱麻。它可以拆成三个层次：平台层负责把操作系统入口、命令行、窗口事件循环接进来； Main 负责跨平台的初始化顺序和运行模式选择；MainLoop 负责真正的“每一帧”。 读这一节时要先记住一个判断：setup/setup2 是“搭环境”，start 是“决定跑什么”， iteration 是“推进一帧”，cleanup 是“按反向顺序拆掉环境”。 标题里的 exe 只是以桌面可执行文件为例；Web、Android、iOS、macOS、Linux 的平台入口文件不同，但都会把控制权交给同一套 Main 初始化和 MainLoop 帧推进逻辑。

主循环一帧里发生什么

物理帧

根据 physics_ticks_per_second 计算固定步长，可能一帧跑多次。每次物理步会先让 SceneTree 做准备，再同步物理 Server、调用 physics_process()、处理导航、执行物理 step()。

空闲帧

RenderingServer 先 sync()，再按窗口是否需要绘制、低功耗模式、资源处理状态决定是否 draw()。

读启动源码时最容易踩的坑

不要把 setup2 当成 SceneTree 创建点

setup2 注册 scene 类型和创建 DisplayServer，但真正的 SceneTree 或自定义 MainLoop 是在 Main::start() 创建并交给 OS::set_main_loop() 的。

不要把平台事件循环和引擎主循环混在一起

OS_Windows::run() 处理窗口消息和调用 Main::iteration()；Main::iteration() 才负责 physics/process/render 的跨平台顺序。

不要只看启动，不看清理

注册顺序是 CORE → SERVERS → SCENE → EDITOR，清理大体反过来。关闭崩溃、热重载问题、扩展卸载问题经常要从 Main::cleanup() 的释放顺序定位。

不要忘记 TOOLS_ENABLED

编辑器、导出、文档工具、项目管理器和很多调试路径只在工具构建中可用。普通 export template 不应依赖 editor-only 类型和单例。

运行时核心：Object、ClassDB、Variant

这一组是 Godot 源码的“运行时语法”。用户看到的节点、资源、Inspector 属性、脚本方法、信号和文档并不是各自独立实现的， 它们共同依赖一套反射和动态调用系统：Object 提供对象身份和通用行为， ClassDB 保存类、方法、属性和信号的元数据，Variant 负责在脚本、编辑器、序列化和 C++ 之间搬运值。 读懂这里，就能解释“为什么一个 C++ 方法能被 GDScript 调用”“为什么 Inspector 自动出现属性”“为什么延迟调用不会立刻改树”。 这里的“反射”可以先理解成一张运行时登记表：C++ 类在编译后本来只对 C++ 编译器清楚，注册到 ClassDB 后，脚本、编辑器、文档生成器和资源系统才能按名字查询它有哪些方法、属性、信号和父类。

一个 C++ 类怎样变成 Godot API

1
类继承 Object 体系并写 GDCLASS

例如 scene/main/node.h:55 的 GDCLASS(Node, Object)。这个宏声明静态类名、父类、类型检查辅助函数和 Godot 运行时需要的类型信息。没有这一步，ClassDB::register_class<T>() 无法正确识别类。

2

注册通常发生在 register_*_types.cpp，通过 GDREGISTER_CLASS 或 ClassDB::register_class 把类放进数据库。宏定义在 core/object/class_db.h:568，真实注册模板入口在 core/object/class_db.h:245。这一步决定脚本能不能按类名实例化对象。

读 Server 源码的三个固定问题

谁拥有真实对象？

查 Server 或 storage 里的 RID_Owner、owner map、storage 类。场景节点通常只保存 RID，不拥有后端对象本体。

调用是立即执行还是排队？

渲染和物理都可能经过 command queue 或 wrap_mt。看到 RenderingServerDefault::sync()、draw()、PhysicsServer3DWrapMT::sync() 时要注意线程边界。

生命周期在哪里结束？

节点退出树或资源释放时通常要调用 Server 的 free_rid 或具体释放函数。RID 泄漏会表现为后端对象还在，场景节点却已经不存在。

渲染：从节点属性到 GPU 命令

渲染源码规模很大，适合按“场景层表示、Server API、Renderer 后端、RenderingDevice、平台显示”五段来读。 不要一开始就读 Vulkan 或 shader 细节，先看 2D/3D 节点如何把 RID、变换、材质、可见性提交给 RenderingServer； 再看 Renderer 如何把这些状态组织成 canvas draw 或 3D scene render；最后才进入 RenderingDevice 和平台 swapchain。 这条顺序很重要：节点解释“用户设置了什么”，Server 解释“状态怎样被提交”，Renderer 解释“这一帧怎样组织绘制”，RenderingDevice 才解释“怎样变成底层图形 API 命令”。 需要注意：下面的 RD 路径主要对应 Forward+/Mobile 等现代渲染后端，底层驱动在 drivers/vulkan、drivers/d3d12、drivers/metal；Compatibility/OpenGL 路径走 drivers/gles3，不要把它也直接归到 RenderingDevice。

资源、Shader 和 RD 的边界

Texture/Image

Image 是 CPU 侧像素数据；Texture Resource 通常持有或生成 RenderingServer texture RID；RenderingDevice texture 是更底层的 GPU 资源。读纹理 bug 时先判断数据错在 CPU Image、Texture Resource、Server RID 还是 GPU texture。

Mesh

Mesh Resource 保存 surface arrays 和材质引用；RenderingServer 的 mesh RID 是提交给渲染后端的表示；3D instance RID 只是“这个 mesh 在某个 scenario 里有一个实例”。

Shader

用户 Shader Resource 先走 Godot shader language 和材质系统，后端再生成具体 shader variant。真正的 RD shader 创建入口在 RenderingDevice，中间还有 shader compiler 和缓存。

Render Thread

RenderingServerDefault::sync() 在 servers/rendering/rendering_server_default.cpp:435，draw() 在 443。多线程渲染时很多调用先进入 command queue，再由渲染线程执行。

物理与导航：抽象 API + 可替换实现

Godot 的物理层也使用 Server/RID 模式。场景层的 CollisionShape、RigidBody、Area、CharacterBody 等节点负责把用户语义转成 PhysicsServer API；真正的 broadphase、窄相碰撞、约束求解由 Godot Physics 或 Jolt 这样的后端实现。读物理源码时不要从“碰撞公式”开始，而要先建立一个更实用的模型：节点保存你在场景里能看懂的状态，Server 保存可求解的 body/area/shape/space，固定物理步负责把两边同步起来。 如果只记三个词：space 是物理世界容器，shape 是碰撞几何，body/area 是带着 shape 进入 space 的物理对象；节点只是把这些对象包装成用户能编辑和脚本能调用的形式。

读物理/导航源码的常见误区

误区一：节点就是底层物理对象

不是。节点是用户 API 和场景树对象，真正被后端求解的是 RID 指向的 body/area/shape/space。调试时要同时看节点属性和 Server 状态。

误区二：所有物理调用都立即执行

多线程包装和固定步会改变执行时机。属性 setter 可能只是提交状态，真正求解在 step()。如果结果晚一帧出现，先查 sync/end_sync/step 的顺序。

误区三：CharacterBody 是 RigidBody 的简化版

CharacterBody 是可控运动器，核心是 motion test 和碰撞响应策略；RigidBody 是求解器积分对象。两者遇到 bug 时阅读路径完全不同。

误区四：导航会帮你避免所有碰撞

导航路径只描述可走路线和局部避障建议，最终移动还要经过 CharacterBody、RigidBody 或自定义物理逻辑。导航和物理要在游戏逻辑层协作。

误区五：2D 和 3D 可以完全照搬

Godot 的 2D/3D 物理接口相似，但 shape 类型、space 实现、后端选择和节点生命周期并不完全相同。读源码时可以类比结构，不能假设细节一致。

音频、输入、GUI、动画：用户体验层的主要系统

这些系统跨越 core、scene、servers 和 editor。它们读起来比渲染/物理更贴近用户 API，但同样遵循“场景节点封装、核心数据结构支撑、Server 或资源系统执行”的模式。建议把这一组系统理解成“玩家和编辑器操作者真正触摸到引擎的地方”：耳朵听到的是 AudioServer 混音后的结果，手指和键盘进入的是 InputEvent，眼睛看到和点击的是 Control/Theme/TextServer 共同排出的界面，动画系统则在每帧把时间变成属性变化。 这些不是一个巨型子系统，而是几条相互交叉的链路；遇到问题时先判断它属于声音输出、输入派发、控件布局、文字排版、动画写属性，还是调试采样。

读脚本和扩展源码的常见误区

误区一：脚本直接修改 Node 类定义

脚本实例挂在 Object 上，不会改变 C++ 类本身。属性、方法、信号是通过 ScriptInstance 和 Object 的反射路径合并出来的。

误区二：GDScript 只是解释器

GDScript 有 Parser、Analyzer、Compiler、ByteCodeGenerator、GDScriptFunction、debugger、profiler、language server。要定位类型错误、补全错误、运行时错误，需要知道错误属于哪一阶段。

误区三：C# 和 GDScript 共享实现细节

它们共享 ScriptLanguage/Script/ScriptInstance/Object 桥，但 C# 的运行时、程序集加载、托管对象句柄和热重载路径完全不同。

误区四：GDExtension 是普通脚本语言

GDExtension 不是 ScriptLanguage；它是原生扩展 ABI。它可以注册类、方法、属性、信号和回调，让动态库像内置 C++ 类一样进入 ClassDB。

误区五：Variant 是“慢但方便”的小工具

Variant 是跨语言、跨编辑器、跨序列化、跨扩展 ABI 的核心边界。脚本系统的很多复杂性都来自“如何在保留类型信息的同时统一调用”。

编辑器：一个用 Godot 自己写出来的大型工具程序

读编辑器源码的常见误区

误区一：EditorNode 什么都做

EditorNode 是调度中心，不是所有业务逻辑所在地。保存场景会进入 PackedScene/ResourceSaver，属性编辑会进入 Inspector/UndoRedo/Object，导入会进入 EditorFileSystem/ResourceImporter，运行会进入 EditorRun。

误区二：编辑器 UI 和游戏 UI 是两套技术

编辑器大量使用 Control、Container、Theme、PopupMenu、Tree 等 scene/gui 组件。读布局、输入、主题、文字问题时，要回到 GUI 系统，而不是只在 editor 目录里找。

误区三：Inspector 手写了每个类的属性面板

大多数属性来自 ClassDB、Object、ScriptInstance 和 PropertyInfo；Inspector 负责把这些元数据转成控件。只有特殊资源或节点才会通过 EditorInspectorPlugin 或专用 EditorPlugin 插入自定义 UI。

误区四：导入只是打开文件时临时转换

编辑器会生成 .import 元数据和内部资源，ResourceLoader 再通过 ResourceFormatImporter 找到实际加载路径。导入状态不清楚时，只看原始 png/fbx/wav 文件是不够的。

误区五：点击运行就是当前 SceneTree 切换到游戏

编辑器通常启动新的 Godot 实例并建立调试连接。要理解运行参数、远程调试、profile、breakpoint，需要同时看 editor/run、editor/debugger 和 core/debugger。

平台、驱动、第三方：把抽象落到真实系统

platform 目录把 Godot 的抽象接口接到真实系统：进程入口、窗口、输入、文件系统、动态库、线程、音频设备、移动端生命周期、Web 浏览器事件都在这里落地。drivers 放跨平台或半跨平台的驱动适配，thirdparty 放 vendored 外部库，modules 再把其中很多库包装成 Godot 功能。读平台相关源码的核心问题不是“Windows 文件在哪里”，而是“某个引擎抽象最终由哪个平台实现提供”。

平台代码最容易踩的判断点

先确认构建目标

SConstruct:533 定义 editor/debug features，SConstruct:703 打印平台、架构、target，SConstruct:1243 只构建选中的 platform/<platform>/SCsub。同一段源码在 editor、template_debug、template_release 下可能行为不同。

再确认启用的 driver

drivers/SCsub 根据 env["vulkan"]、env["d3d12"]、env["opengl3"]、env["metal"]、env["sdl"] 决定编译哪些驱动。运行时能注册不代表当前二进制一定包含。

不要把 thirdparty 当作 Godot 业务层

如果渲染 shader 编译出错，先看 Godot 的 shader compiler、RenderingDevice 和 glslang 包装层；如果字体布局出错，先看 TextServer、freetype/harfbuzz 模块边界。只有确认是外部库行为时再深入 thirdparty。

桌面、移动、Web 的事件循环模型不同

Windows/Linux 常见是 OS::run() 阻塞循环；Android 每帧由 Java/JNI step 驱动；Web 由浏览器 animation frame 驱动。定位“为什么没进一帧”时必须先确认谁拥有主循环。

平台层不应该知道太多场景细节

健康边界是平台层产出窗口、输入、文件、音频、动态库、时间等能力；场景树、节点生命周期、脚本回调应该在 core/scene/main 里完成。如果平台实现直接操作高层对象，要格外留意耦合和线程问题。

平台代码和驱动代码最容易出现条件编译。读这类文件时，先确认当前构建宏：目标平台、工具构建、渲染驱动、启用模块、禁用 3D、线程模式，否则很容易误判某段代码是否会参与当前二进制。

2. ObjectID 只能重新查找对象

ObjectDB::get_instance() 在 core/object/object.h:908。它能把 ObjectID 找回成当前仍然活着的 Object 指针，但不会阻止对象释放。需要跨帧保存对象时，必须设计好强引用、弱引用或重新校验对象是否还在。

3. 析构时从 ObjectDB 移除

Object::~Object() 在 core/object/object.cpp:2320，会在 2353 到 2355 移除 ObjectDB 记录。预删除通知在 core/object/object.cpp:173 和 179，这解释了为什么释放对象时仍会触发最后一轮清理通知。

memnew / memdelete：Godot 自己的分配钩子

Godot 大量使用 memnew 和 memdelete，不是普通风格偏好，而是为了接入统一内存统计、对象 post-initialize、pre-delete 和 debug 检查。宏定义在 core/os/memory.h:150 和 160，底层分配函数是 Memory::alloc_static() 的 core/os/memory.cpp:82，释放是 Memory::free_static() 的 173。对 Object 来说，memdelete 会先进入 predelete_handler()，再析构并释放内存；Object 的 predelete/postinitialize 钩子在 core/object/object.cpp:2373 和 2377。

线程边界：为什么有 WorkerThreadPool、process thread group 和 wrap_mt

WorkerThreadPool 是通用任务池

类入口在 core/object/worker_thread_pool.h:44。普通任务入口是 WorkerThreadPool::add_task() 的 core/object/worker_thread_pool.cpp:385，组任务入口是 add_group_task() 的 706，等待组任务完成在 727。资源加载、场景处理、某些批处理会用它分摊工作。

Node process thread group 是场景层并行边界

Node::set_process_thread_group() 在 scene/main/node.cpp:1203，SceneTree 处理分组的线程任务在 scene/main/scene_tree.cpp:1286 到 1313。它不是让所有 Node 都随意跨线程访问场景树，而是把特定 process group 安排到受控线程阶段。

call_deferred_thread_group 是线程组内消息边界

绑定入口在 scene/main/node.cpp:3914，实现入口在 scene/main/node.cpp:3673。如果某个节点在子线程 process group 中运行，消息必须进入对应组的队列和安全刷新点。

wrap_mt 是 Server 调用的线程隔离包装

物理的多线程包装文件在 servers/physics_2d/physics_server_2d_wrap_mt.* 和 servers/physics_3d/physics_server_3d_wrap_mt.*，公共机制在 servers/server_wrap_mt_common.h。读这类代码时要区分 API 调用者看到的同步接口，和内部实际排队、flush、sync、step 的执行时机。

ResourceCache：为什么同一路径加载出来可能是同一个对象

ResourceLoader::load() 入口在 core/io/resource_loader.cpp:513，内部会根据缓存模式查 ResourceCache。缓存表在 core/io/resource.cpp:804，按路径取 Ref 的入口是 ResourceCache::get_ref() 的 core/io/resource.cpp:852。资源设置路径时会更新缓存，相关逻辑在 core/io/resource.cpp:89 到 115。所以“我加载两次资源为什么状态共享”通常不是 bug，而是缓存模式和资源可变性共同作用。

错误处理风格

Godot 大量使用 ERR_FAIL_COND*、ERR_FAIL_NULL*、ERR_PRINT* 这类宏。它们让源码保持早返回风格，也让 debug/editor 构建能提供更完整的错误信息。不要把这些宏简单看成 assert；很多会在 release 中保留行为。读 bug 时要注意宏后面的返回值、是否会继续执行、是否只打印警告。

为什么生成代码这么多

引擎需要把 C++ 类型、脚本 API、GDExtension API、文档、shader、图标、平台类路径和模块宏统一到构建产物里。生成代码降低手工同步成本，但也要求读源码时分清“源模板”和“生成结果”。例如模块总入口来自 modules/modules_builders.py:15，不是手写的 modules/register_module_types.gen.cpp；类文档、图标、shader 和 extension API 也有各自的生成器。

每条路线的结束标准

阅读阶段能回答的问题应该产出的笔记还不该做什么
入门路线Godot 从平台入口到 Main、SceneTree、Node、Server、EditorNode 大概怎么串起来？一张目录依赖图、一张启动流程图、一个主循环阶段表。不要急着改底层内存、线程、资源格式或 Server 后端。
机制路线某个 API、节点、资源或编辑器按钮最终落到哪些类和函数？一条端到端调用链：用户操作/API → 绑定 → 对象状态 → Resource/Server → 主循环阶段。不要只改可见 bug 点；必须确认序列化、Inspector、导入导出和测试是否受影响。
深入路线这个改动会不会破坏生命周期、缓存、线程边界、ABI 或生成代码？影响面清单、回归测试入口、构建配置差异、失败回滚点。不要在没有验证多平台/多构建目标的情况下合并平台层或 core 层改动。

建议的“功能追踪法”

从用户 API 开始

先找文档类名或节点类名，例如 Sprite2D、RigidBody3D、AnimationPlayer。入口通常在 scene、modules 或 editor，不要从 backend 细节开始。

找绑定

搜索 _bind_methods()，看属性、方法、信号如何暴露给脚本、ClassDB、Inspector 和文档。绑定表能告诉你哪些行为是公开 API，哪些只是内部辅助。

找内部状态

看头文件成员变量，理解这个类真正保存了什么状态，哪些只是转发给 Server，哪些是缓存，哪些需要序列化。不要只读函数体。

找 Resource / Server 边界

如果出现 ResourceLoader、ResourceSaver、RID、RenderingServer、PhysicsServer，就沿着资源格式或 Server API 继续追，确认真实数据在哪里。

回到主循环

最后确认这个状态在哪个阶段被刷新：process、physics_process、sync、draw、flush、import、export。很多“为什么现在没生效”的答案就在同步点。

写下影响面再改

至少检查：脚本 API、C++ ABI/GDExtension、保存格式、Inspector、导入缓存、导出模板、线程边界、测试。越靠近 core/servers/platform，影响面越要写清楚。
