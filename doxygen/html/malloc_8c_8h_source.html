14 * Quickstart
15 
16  This library is all in one file to simplify the most common usage:
17  ftp it, compile it (-O3), and link it into another program. All of
18  the compile-time options default to reasonable values for use on
19  most platforms. You might later want to step through various
20  compile-time and dynamic tuning options.

36 * Vital statistics:
37 
38  Supported pointer/size_t representation: 4 or 8 bytes
39  size_t MUST be an unsigned type of the same width as
40  pointers. (If you are using an ancient system that declares
41  size_t as a signed type, or need it to be a different width
42  than pointers, you can use a previous release of this malloc
43  (e.g. 2.7.2) supporting these.)
44 
45  Alignment: 8 bytes (default)
46  This suffices for nearly all current machines and C compilers.
47  However, you can define MALLOC_ALIGNMENT to be wider than this
48  if necessary (up to 128bytes), at the expense of using more space.
49 
50  Minimum overhead per allocated chunk: 4 or 8 bytes (if 4byte sizes)
51  8 or 16 bytes (if 8byte sizes)
52  Each malloced chunk has a hidden word of overhead holding size
53  and status information, and additional cross-check word
54  if FOOTERS is defined.
55 
56  Minimum allocated size: 4-byte ptrs: 16 bytes (including overhead)
57  8-byte ptrs: 32 bytes (including overhead)
58 
59  Even a request for zero bytes (i.e., malloc(0)) returns a
60  pointer to something of the minimum allocatable size.
61  The maximum overhead wastage (i.e., number of extra bytes
62  allocated than were requested in malloc) is less than or equal
63  to the minimum size, except for requests >= mmap_threshold that
64  are serviced via mmap(), where the worst case wastage is about
65  32 bytes plus the remainder from a system page (the minimal
66  mmap unit); typically 4096 or 8192 bytes.
67 
68  Security: static-safe; optionally more or less
69  The "security" of malloc refers to the ability of malicious
70  code to accentuate the effects of errors (for example, freeing
71  space that is not currently malloc'ed or overwriting past the
72  ends of chunks) in code that calls malloc. This malloc
73  guarantees not to modify any memory locations below the base of
74  heap, i.e., static variables, even in the presence of usage
75  errors. The routines additionally detect most improper frees
76  and reallocs. All this holds as long as the static bookkeeping
77  for malloc itself is not corrupted by some other means. This
78  is only one aspect of security -- these checks do not, and
79  cannot, detect all possible programming errors.
80 
81  If FOOTERS is defined nonzero, then each allocated chunk
82  carries an additional check word to verify that it was malloced
83  from its space. These check words are the same within each
84  execution of a program using malloc, but differ across
85  executions, so externally crafted fake chunks cannot be
86  freed. This improves security by rejecting frees/reallocs that
87  could corrupt heap memory, in addition to the checks preventing
88  writes to statics that are always on. This may further improve
89  security at the expense of time and space overhead. (Note that
90  FOOTERS may also be worth using with MSPACES.)
91 
92  By default detected errors cause the program to abort (calling
93  "abort()"). You can override this to instead proceed past
94  errors by defining PROCEED_ON_ERROR. In this case, a bad free
95  has no effect, and a malloc that encounters a bad address
96  caused by user overwrites will ignore the bad address by
97  dropping pointers and indices to all known memory. This may
98  be appropriate for programs that should continue if at all
99  possible in the face of programming errors, although they may
100  run out of memory because dropped memory is never reclaimed.
101 
102  If you don't like either of these options, you can define
103  CORRUPTION_ERROR_ACTION and USAGE_ERROR_ACTION to do anything
104  else. And if if you are sure that your program using malloc has
105  no errors or vulnerabilities, you can define INSECURE to 1,
106  which might (or might not) provide a small performance improvement.
107 
108  Thread-safety: NOT thread-safe unless USE_LOCKS defined
109  When USE_LOCKS is defined, each public call to malloc, free,
110  etc is surrounded with either a pthread mutex or a win32
111  spinlock (depending on WIN32). This is not especially fast, and
112  can be a major bottleneck. It is designed only to provide
113  minimal protection in concurrent environments, and to provide a
114  basis for extensions. If you are using malloc in a concurrent
115  program, consider instead using nedmalloc
116  (http://www.nedprod.com/programs/portable/nedmalloc/) or
117  ptmalloc (See http://www.malloc.de), which are derived
118  from versions of this malloc.
119 
120  System requirements: Any combination of MORECORE and/or MMAP/MUNMAP
121  This malloc can use unix sbrk or any emulation (invoked using
122  the CALL_MORECORE macro) and/or mmap/munmap or any emulation
123  (invoked using CALL_MMAP/CALL_MUNMAP) to get and release system
124  memory. On most unix systems, it tends to work best if both
125  MORECORE and MMAP are enabled. On Win32, it uses emulations
126  based on VirtualAlloc. It also uses common C library functions
127  like memset.
128 
129  Compliance: I believe it is compliant with the Single Unix Specification
130  (See http://www.unix.org). Also SVID/XPG, ANSI C, and probably
131  others as well.

133 * Overview of algorithms
134 
135  This is not the fastest, most space-conserving, most portable, or
136  most tunable malloc ever written. However it is among the fastest
137  while also being among the most space-conserving, portable and
138  tunable. Consistent balance across these factors results in a good
139  general-purpose allocator for malloc-intensive programs.
140 
141  In most ways, this malloc is a best-fit allocator. Generally, it
142  chooses the best-fitting existing chunk for a request, with ties
143  broken in approximately least-recently-used order. (This strategy
144  normally maintains low fragmentation.) However, for requests less
145  than 256bytes, it deviates from best-fit when there is not an
146  exactly fitting available chunk by preferring to use space adjacent
147  to that used for the previous small request, as well as by breaking
148  ties in approximately most-recently-used order. (These enhance
149  locality of series of small allocations.) And for very large requests
150  (>= 256Kb by default), it relies on system memory mapping
151  facilities, if supported. (This helps avoid carrying around and
152  possibly fragmenting memory used only for large chunks.)
153 
154  All operations (except malloc_stats and mallinfo) have execution
155  times that are bounded by a constant factor of the number of bits in
156  a size_t, not counting any clearing in calloc or copying in realloc,
157  or actions surrounding MORECORE and MMAP that have times
158  proportional to the number of non-contiguous regions returned by
159  system allocation routines, which is often just 1. In real-time
160  applications, you can optionally suppress segment traversals using
161  NO_SEGMENT_TRAVERSAL, which assures bounded execution even when
162  system allocators return non-contiguous spaces, at the typical
163  expense of carrying around more memory and increased fragmentation.
164 
165  The implementation is not very modular and seriously overuses
166  macros. Perhaps someday all C compilers will do as good a job
167  inlining modular code as can now be done by brute-force expansion,
168  but now, enough of them seem not to.
169 
170  Some compilers issue a lot of warnings about code that is
171  dead/unreachable only on some platforms, and also about intentional
172  uses of negation on unsigned types. All known cases of each can be
173  ignored.
174 
175  For a longer but out of date high-level description, see
176  http://gee.cs.oswego.edu/dl/html/malloc.html

178 * MSPACES
179  If MSPACES is defined, then in addition to malloc, free, etc.,
180  this file also defines mspace_malloc, mspace_free, etc. These
181  are versions of malloc routines that take an "mspace" argument
182  obtained using create_mspace, to control all internal bookkeeping.
183  If ONLY_MSPACES is defined, only these versions are compiled.
184  So if you would like to use this allocator for only some allocations,
185  and your system malloc for others, you can compile with
186  ONLY_MSPACES and then do something like...
187  static mspace mymspace = create_mspace(0,0); // for example
188  #define mymalloc(bytes) mspace_malloc(mymspace, bytes)
189 
190  (Note: If you only need one instance of an mspace, you can instead
191  use "USE_DL_PREFIX" to relabel the global malloc.)
192 
193  You can similarly create thread-local allocators by storing
194  mspaces as thread-locals. For example:
195  static __thread mspace tlms = 0;
196  void* tlmalloc(size_t bytes) {
197  if (tlms == 0) tlms = create_mspace(0, 0);
198  return mspace_malloc(tlms, bytes);
199  }
200  void tlfree(void* mem) { mspace_free(tlms, mem); }
201 
202  Unless FOOTERS is defined, each mspace is completely independent.
203  You cannot allocate from one and free to another (although
204  conformance is only weakly checked, so usage errors are not always
205  caught). If FOOTERS is defined, then each chunk carries around a tag
206  indicating its originating mspace, and frees are directed to their
207  originating spaces.

210 
211 Be careful in setting #define values for numerical constants of type

216 WIN32 default: defined if _WIN32 defined
217  Defining WIN32 sets up defaults for MS environment and compilers.
218  Otherwise defaults are for unix. Beware that there seem to be some
219  cases where this malloc might not be a pure drop-in replacement for
220  Win32 malloc: Random-looking failures from Win32 GDI API's (eg;
221  SetDIBits()) may be due to bugs in some video driver implementations
222  when pixel buffers are malloc()ed, and the region spans more than
223  one VirtualAlloc()ed region. Because dlmalloc uses a small (64Kb)
224  default granularity, pixel buffers may straddle virtual allocation
225  regions more often than when using the Microsoft allocator. You can
226  avoid this by using VirtualAlloc() and VirtualFree() for all pixel
227  buffers rather than using malloc(). If this is not possible,
228  recompile this malloc with a larger DEFAULT_GRANULARITY.

231  Controls the minimum alignment for malloc'ed chunks. It must be a
232  power of two and at least 8, even on machines for which smaller
233  alignments would suffice. It may be defined as larger than this
234  though. Note however that code and data structures are optimized for

244 USE_LOCKS default: 0 (false)
245  Causes each call to each public routine to be surrounded with
246  pthread or WIN32 mutex lock/unlock. (If set true, this can be
247  overridden on a per-mspace basis for mspace versions.) If set to a
248  non-zero value other than 1, locks are used, but their
249  implementation is left out, so lock functions must be supplied manually,
250  as described below.

252 USE_SPIN_LOCKS default: 1 iff USE_LOCKS and on x86 using gcc or MSC
253  If true, uses custom spin locks for locking. This is currently
254  supported only for x86 platforms using gcc or recent MS compilers.
255  Otherwise, posix locks or win32 critical sections are used.
256 
257 FOOTERS default: 0
258  If true, provide extra checking and dispatching by placing
259  information in the footers of allocated chunks. This adds
260  space and time overhead.
261 
262 INSECURE default: 0
263  If true, omit checks for usage errors and heap space overwrites.
264 
265 USE_DL_PREFIX default: NOT defined
266  Causes compiler to prefix all public routines with the string 'dl'.
267  This can be useful when you only want to use this malloc in one part
268  of a program, using your regular system malloc elsewhere.
269 
270 ABORT default: defined as abort()
271  Defines how to abort on failed checks. On most systems, a failed
272  check cannot die with an "assert" or even print an informative
273  message, because the underlying print routines in turn call malloc,
274  which will fail again. Generally, the best policy is to simply call
275  abort(). It's not very useful to do more than this because many
276  errors due to overwriting will show up as address faults (null, odd
277  addresses etc) rather than malloc-triggered checks, so will also
278  abort. Also, most compilers know that abort() does not return, so
279  can better optimize code conditionally calling it.
280 
281 PROCEED_ON_ERROR default: defined as 0 (false)
282  Controls whether detected bad addresses cause them to bypassed
283  rather than aborting. If set, detected bad arguments to free and
284  realloc are ignored. And all bookkeeping information is zeroed out
285  upon a detected overwrite of freed heap space, thus losing the
286  ability to ever return it from malloc again, but enabling the
287  application to proceed. If PROCEED_ON_ERROR is defined, the
288  static variable malloc_corruption_error_count is compiled in
289  and can be examined to see if errors have occurred. This option
290  generates slower code than the default abort policy.
291 
292 DEBUG default: NOT defined
293  The DEBUG setting is mainly intended for people trying to modify
294  this code or diagnose problems when porting to new platforms.
295  However, it may also be able to better isolate user errors than just
296  using runtime checks. The assertions in the check routines spell
297  out in more detail the assumptions and invariants underlying the
298  algorithms. The checking is fairly extensive, and will slow down
299  execution noticeably. Calling malloc_stats or mallinfo with DEBUG
300  set will attempt to check every non-mmapped allocated and free chunk
301  in the course of computing the summaries.
302 
303 ABORT_ON_ASSERT_FAILURE default: defined as 1 (true)
304  Debugging assertion failures can be nearly impossible if your
305  version of the assert macro causes malloc to be called, which will
306  lead to a cascade of further failures, blowing the runtime stack.
307  ABORT_ON_ASSERT_FAILURE cause assertions failures to call abort(),
308  which will usually make debugging easier.
309 
310 MALLOC_FAILURE_ACTION default: sets errno to ENOMEM, or no-op on win32
311  The action to take before "return 0" when malloc fails to be able to
312  return memory because there is none available.

314 HAVE_MORECORE default: 1 (true) unless win32 or ONLY_MSPACES
315  True if this system supports sbrk or an emulation of it.
316 
317 MORECORE default: sbrk
318  The name of the sbrk-style system routine to call to obtain more
319  memory. See below for guidance on writing custom MORECORE
320  functions. The type of the argument to sbrk/MORECORE varies across
321  systems. It cannot be size_t, because it supports negative
322  arguments, so it is normally the signed type of the same width as
323  size_t (sometimes declared as "intptr_t"). It doesn't much matter
324  though. Internally, we only call it with arguments less than half
325  the max value of a size_t, which should work across all reasonable
326  possibilities, although sometimes generating compiler warnings.
327 
328 MORECORE_CONTIGUOUS default: 1 (true) if HAVE_MORECORE

330  with positive arguments always return contiguous increasing
331  addresses. This is true of unix sbrk. It does not hurt too much to
332  set it true anyway, since malloc copes with non-contiguities.
333  Setting it false when definitely non-contiguous saves time
334  and possibly wasted space it would take to discover this though.

343  If non-zero, suppresses traversals of memory segments
344  returned by either MORECORE or CALL_MMAP. This disables

349  True if this system supports mmap or an emulation of it. If so, and

443 DEFAULT_TRIM_THRESHOLD default: 2MB
444  Also settable using mallopt(M_TRIM_THRESHOLD, x)
445  The maximum amount of unused top-most memory to keep before
446  releasing via malloc_trim in free(). Automatic trimming is mainly
447  useful in long-lived programs using contiguous MORECORE. Because
448  trimming via sbrk can be slow on some systems, and can sometimes be
449  wasteful (in cases where programs immediately afterward allocate
450  more large chunks) the value should be high enough so that your
451  overall system performance would improve by releasing this much
452  memory. As a rough guide, you might set to a value close to the
453  average size of a process (program) running on your system.
454  Releasing this much memory would allow such a process to run in
455  memory. Generally, it is worth tuning trim thresholds when a
456  program undergoes phases where several large chunks are allocated
457  and released in ways that can reuse each other's storage, perhaps
458  mixed with phases where there are no such chunks at all. The trim
459  value must be greater than page size to have any useful effect. To
460  disable trimming completely, you can set to MAX_SIZE_T. Note that the trick
461  some people use of mallocing a huge space and then freeing it at
462  program startup, in an attempt to reserve system memory, doesn't
463  have the intended effect under automatic trimming, since that memory
464  will immediately be returned to the system.
465 
466 DEFAULT_MMAP_THRESHOLD default: 256K
467  Also settable using mallopt(M_MMAP_THRESHOLD, x)
468  The request size threshold for using MMAP to directly service a
469  request. Requests of at least this size that cannot be allocated
470  using already-existing space will be serviced via mmap. (If enough
471  normal freed space already exists it is used instead.) Using mmap
472  segregates relatively large chunks of memory so that they can be
473  individually obtained and released from the host system. A request
474  serviced through mmap is never reused by any other request (at least
475  not directly; the system may just so happen to remap successive
476  requests to the same locations). Segregating space in this way has
477  the benefits that: Mmapped space can always be individually released
478  back to the system, which helps keep the system level memory demands
479  of a long-lived program low. Also, mapped memory doesn't become
480  `locked' between other chunks, as can happen with normally allocated
481  chunks, which means that even trimming via malloc_trim would not
482  release them. However, it has the disadvantage that the space
483  cannot be reclaimed, consolidated, and then used to service later
484  requests, as happens with normal chunks. The advantages of mmap
485  nearly always outweigh disadvantages for "large" chunks, but the
486  value of "large" may vary across systems. The default is an
487  empirically derived value that works well in most systems. You can
488  disable mmap by setting to MAX_SIZE_T.
489 
490 MAX_RELEASE_CHECK_RATE default: 4095 unless not HAVE_MMAP
491  The number of consolidated frees between checks to release
492  unused segments when freeing. When using non-contiguous segments,
493  especially with multiple mspaces, checking only for topmost space
494  doesn't always suffice to trigger trimming. To compensate for this,
495  free() will, with a period of MAX_RELEASE_CHECK_RATE (or the
496  current number of segments, if greater) try to release unused
497  segments to the OS when freeing chunks that result in
498  consolidation. The best value for this parameter is a compromise
499  between slowing down frees with relatively costly checks that
500  rarely trigger versus holding on to unused memory. To effectively
501  disable, set to MAX_SIZE_T. This may lead to a very slight speed
502  improvement at the expense of carrying around more memory.

2036 
2037 /*
2038  (The following includes lightly edited explanations by Colin Plumb.)
2039 
2040  The malloc_chunk declaration below is misleading (but accurate and
2041  necessary). It declares a "view" into memory allowing access to
2042  necessary fields at known offsets from a given base.
2043 
2044  Chunks of memory are maintained using a `boundary tag' method as
2045  originally described by Knuth. (See the paper by Paul Wilson
2046  ftp://ftp.cs.utexas.edu/pub/garbage/allocsrv.ps for a survey of such
2047  techniques.) Sizes of free chunks are stored both in the front of
2048  each chunk and at the end. This makes consolidating fragmented
2049  chunks into bigger chunks fast. The head fields also hold bits
2050  representing whether chunks are free or in use.
2051 
2052  Here are some pictures to make it clearer. They are "exploded" to
2053  show that the state of a chunk can be thought of as extending from
2054  the high 31 bits of the head field of its header through the
2055  prev_foot and PINUSE_BIT bit of the following chunk header.
2056 
2057  A chunk that's in use looks like:
2058 
2059  chunk-> +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2060  | Size of previous chunk (if P = 0) |
2061  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2062  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+ |P|
2063  | Size of this chunk 1| +-+
2064  mem-> +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2065  | |
2066  +- -+
2067  | |
2068  +- -+
2069  | :
2070  +- size - sizeof(size_t) available payload bytes -+
2071  : |
2072  chunk-> +- -+
2073  | |
2074  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2075  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+ |1|
2076  | Size of next chunk (may or may not be in use) | +-+
2077  mem-> +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2078 
2079  And if it's free, it looks like this:
2080 
2081  chunk-> +- -+
2082  | User payload (must be in use, or we would have merged!) |
2083  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2084  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+ |P|
2085  | Size of this chunk 0| +-+
2086  mem-> +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2087  | Next pointer |
2088  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2089  | Prev pointer |
2090  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2091  | :
2092  +- size - sizeof(struct chunk) unused bytes -+
2093  : |
2094  chunk-> +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2095  | Size of this chunk |
2096  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2097  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+ |0|
2098  | Size of next chunk (must be in use, or we would have merged)| +-+
2099  mem-> +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2100  | :
2101  +- User payload -+
2102  : |
2103  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2104  |0|
2105  +-+
2106  Note that since we always merge adjacent free chunks, the chunks
2107  adjacent to a free chunk must be in use.
2108 
2109  Given a pointer to a chunk (which can be derived trivially from the
2110  payload pointer) we can, in O(1) time, find out whether the adjacent
2111  chunks are free, and if so, unlink them from the lists that they
2112  are on and merge them with the current chunk.
2113 
2114  Chunks always begin on even word boundaries, so the mem portion
2115  (which is returned to the user) is also on an even word boundary, and
2116  thus at least double-word aligned.
2117 
2118  The P (PINUSE_BIT) bit, stored in the unused low-order bit of the
2119  chunk size (which is always a multiple of two words), is an in-use
2120  bit for the *previous* chunk. If that bit is *clear*, then the
2121  word before the current chunk size contains the previous chunk
2122  size, and can be used to find the front of the previous chunk.
2123  The very first chunk allocated always has this bit set, preventing
2124  access to non-existent (or non-owned) memory. If pinuse is set for
2125  any given chunk, then you CANNOT determine the size of the
2126  previous chunk, and might even get a memory addressing fault when
2127  trying to do so.
2128 
2129  The C (CINUSE_BIT) bit, stored in the unused second-lowest bit of
2130  the chunk size redundantly records whether the current chunk is
2131  inuse (unless the chunk is mmapped). This redundancy enables usage
2132  checks within free and realloc, and reduces indirection when freeing
2133  and consolidating chunks.
2134 
2135  Each freshly allocated chunk must have both cinuse and pinuse set.
2136  That is, each allocated chunk borders either a previously allocated
2137  and still in-use chunk, or the base of its memory arena. This is
2138  ensured by making all allocations from the the `lowest' part of any
2139  found chunk. Further, no free chunk physically borders another one,
2140  so each free chunk is known to be preceded and followed by either
2141  inuse chunks or the ends of memory.
2142 
2143  Note that the `foot' of the current chunk is actually represented
2144  as the prev_foot of the NEXT chunk. This makes it easier to
2145  deal with alignments etc but can be very confusing when trying
2146  to extend or adapt this code.
2147 
2148  The exceptions to all this are
2149 
2150  1. The special chunk `top' is the top-most available chunk (i.e.,
2151  the one bordering the end of available memory). It is treated
2152  specially. Top is never included in any bin, is used only if
2153  no other chunk is available, and is released back to the
2154  system if it is very large (see M_TRIM_THRESHOLD). In effect,
2155  the top chunk is treated as larger (and thus less well
2156  fitting) than any other available chunk. The top chunk
2157  doesn't update its trailing size field since there is no next
2158  contiguous chunk that would have to index off it. However,
2159  space is still allocated for it (TOP_FOOT_SIZE) to enable
2160  separation or merging when space is extended.
2161 
2162  3. Chunks allocated via mmap, have both cinuse and pinuse bits
2163  cleared in their head fields. Because they are allocated
2164  one-by-one, each must carry its own prev_foot field, which is
2165  also used to hold the offset this chunk has within its mmapped
2166  region, which is needed to preserve alignment. Each mmapped
2167  chunk is trailed by the first two fields of a fake next-chunk
2168  for sake of usage checks.
2169 
2170 */

2311  Larger chunks are kept in a form of bitwise digital trees (aka
2312  tries) keyed on chunksizes. Because malloc_tree_chunks are only for
2313  free chunks greater than 256 bytes, their size doesn't impose any
2314  constraints on user chunk sizes. Each node looks like:
2315 
2316  chunk-> +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2317  | Size of previous chunk |
2318  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2319  `head:' | Size of chunk, in bytes |P|
2320  mem-> +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2321  | Forward pointer to next chunk of same size |
2322  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2323  | Back pointer to previous chunk of same size |
2324  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2325  | Pointer to left child (child[0]) |
2326  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2327  | Pointer to right child (child[1]) |
2328  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2329  | Pointer to parent |
2330  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2331  | bin index of this chunk |
2332  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2333  | Unused space .
2334  . |
2335 nextchunk-> +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2336  `foot:' | Size of chunk, in bytes |
2337  +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
2338 
2339  Each tree holding treenodes is a tree of unique chunk sizes. Chunks
2340  of the same size are arranged in a circularly-linked list, with only
2341  the oldest chunk (the next to be used, in our FIFO ordering)
2342  actually in the tree. (Tree members are distinguished by a non-null
2343  parent pointer.) If a chunk with the same size an an existing node
2344  is inserted, it is linked off the existing node using pointers that
2345  work in the same way as fd/bk pointers of small chunks.
2346 
2347  Each tree contains a power of 2 sized range of chunk sizes (the
2348  smallest is 0x100 <= x < 0x180), which is is divided in half at each
2349  tree level, with the chunks in the smaller half of the range (0x100
2350  <= x < 0x140 for the top nose) in the left subtree and the larger
2351  half (0x140 <= x < 0x180) in the right subtree. This is, of course,
2352  done by inspecting individual bits.
2353 
2354  Using these rules, each node's left subtree contains all smaller
2355  sizes than its right subtree. However, the node at the root of each
2356  subtree has no particular ordering relationship to either. (The
2357  dividing line between the subtree sizes is based on trie relation.)
2358  If we remove the last chunk of a given size from the interior of the
2359  tree, we need to replace it with a leaf node. The tree ordering
2360  rules permit a node to be replaced by any leaf below it.
2361 
2362  The smallest chunk in a tree (a common operation in a best-fit
2363  allocator) can be found by walking a path to the leftmost leaf in
2364  the tree. Unlike a usual binary tree, where we follow left child
2365  pointers until we reach a null, here we follow the right child
2366  pointer any time the left one is null, until we reach a leaf with
2367  both child pointers null. The smallest chunk in the tree will be
2368  somewhere along that path.
2369 
2370  The worst case number of steps to add, find, or remove a node is
2371  bounded by the number of bits differentiating chunks within
2372  bins. Under current bin calculations, this ranges from 6 up to 21
2373  (for 32 bit sizes) or up to 53 (for 64 bit sizes). The typical case
2374  is of course much better.
2375 */

2398 /*
2399  Each malloc space may include non-contiguous segments, held in a
2400  list headed by an embedded malloc_segment record representing the
2401  top-most space. Segments also include flags holding properties of
2402  the space. Large chunks that are directly allocated by mmap are not
2403  included in this list. They are instead independently created and
2404  destroyed without otherwise keeping track of them.
2405 
2406  Segment management mainly comes into play for spaces allocated by
2407  MMAP. Any call to MMAP might or might not return memory that is
2408  adjacent to an existing segment. MORECORE normally contiguously
2409  extends the current space, so this space is almost always adjacent,
2410  which is simpler and faster to deal with. (This is why MORECORE is
2411  used preferentially to MMAP when both are available -- see
2412  sys_alloc.) When allocating using MMAP, we don't use any of the
2413  hinting mechanisms (inconsistently) supported in various
2414  implementations of unix mmap, or distinguish reserving from
2415  committing memory. Instead, we just ask for space, and exploit
2416  contiguity when we get it. It is probably possible to do
2417  better than this on some systems, but no general scheme seems
2418  to be significantly better.
2419 
2420  Management entails a simpler variant of the consolidation scheme
2421  used for chunks to reduce fragmentation -- new adjacent memory is
2422  normally prepended or appended to an existing segment. However,
2423  there are limitations compared to chunk consolidation that mostly
2424  reflect the fact that segment processing is relatively infrequent
2425  (occurring only when getting memory from system) and that we
2426  don't expect to have huge numbers of segments:
2427 
2428  * Segments are not indexed, so traversal requires linear scans. (It
2429  would be possible to index these, but is not worth the extra
2430  overhead and complexity for most programs on most platforms.)
2431  * New segments are only appended to old ones when holding top-most
2432  memory; if they cannot be prepended to others, they are held in
2433  different segments.
2434 
2435  Except for the top-most segment of an mstate, each segment record
2436  is kept at the tail of its segment. Segments are added by pushing
2437  segment records onto the list headed by &mstate.seg for the
2438  containing mstate.
2439 
2440  Segment flags control allocation/merge/deallocation policies:
2441  * If EXTERN_BIT set, then we did not allocate this segment,
2442  and so should not try to deallocate or merge with others.
2443  (This currently holds only for the initial segment passed
2444  into create_mspace_with_base.)
2445  * If USE_MMAP_BIT set, the segment may be merged with
2446  other surrounding mmapped segments and trimmed/de-allocated
2447  using munmap.
2448  * If neither bit is set, then the segment was obtained using
2449  MORECORE so can be merged with surrounding MORECORE'd segments
2450  and deallocated/trimmed using MORECORE with negative arguments.
2451 */

2468 /*
2469  A malloc_state holds all of the bookkeeping for a space.
2470  The main fields are:
2471 
2472  Top
2473  The topmost chunk of the currently active segment. Its size is
2474  cached in topsize. The actual size of topmost space is
2475  topsize+TOP_FOOT_SIZE, which includes space reserved for adding
2476  fenceposts and segment records if necessary when getting more
2477  space from the system. The size at which to autotrim top is
2478  cached from mparams in trim_check, except that it is disabled if
2479  an autotrim fails.
2480

2482  This is the preferred chunk for servicing small requests that
2483  don't have exact fits. It is normally the chunk split off most
2484  recently to service another small request. Its size is cached in
2485  dvsize. The link fields of this chunk are not maintained since it
2486  is not kept in a bin.
2487 
2488  SmallBins
2489  An array of bin headers for free chunks. These bins hold chunks
2490  with sizes less than MIN_LARGE_SIZE bytes. Each bin contains
2491  chunks of all the same size, spaced 8 bytes apart. To simplify
2492  use in double-linked lists, each bin header acts as a malloc_chunk
2493  pointing to the real first node, if it exists (else pointing to
2494  itself). This avoids special-casing for headers. But to avoid
2495  waste, we allocate only the fd/bk pointers of bins, and then use
2496  repositioning tricks to treat these as the fields of a chunk.
2497

2499  Treebins are pointers to the roots of trees holding a range of
2500  sizes. There are 2 equally spaced treebins for each power of two
2501  from TREE_SHIFT to TREE_SHIFT+16. The last bin holds anything
2502  larger.
2503 
2504  Bin maps
2505  There is one bit map for small bins ("smallmap") and one for
2506  treebins ("treemap). Each bin sets its bit when non-empty, and
2507  clears the bit when empty. Bit operations are then used to avoid
2508  bin-by-bin searching -- nearly all "search" is done without ever
2509  looking at bins that won't be selected. The bit maps
2510  conservatively use 32 bits per map word, even if on 64bit system.
2511  For a good description of some of the bit-based techniques used
2512  here, see Henry S. Warren Jr's book "Hacker's Delight" (and
2513  supplement at http://hackersdelight.org/). Many of these are
2514  intended to reduce the branchiness of paths through malloc etc, as
2515  well as to reduce the number of memory locations read or written.
2516 
2517  Segments
2518  A list of segments headed by an embedded malloc_segment record
2519  representing the initial space.
2520 
2521  Address check support
2522  The least_addr field is the least address ever obtained from
2523  MORECORE or MMAP. Attempted frees and reallocs of any address less
2524  than this are trapped (unless INSECURE is defined).
2525 
2526  Magic tag

2528 
2529  Flags
2530  Bits recording whether to use MMAP, locks, or contiguous MORECORE
2531 
2532  Statistics
2533  Each space keeps track of current and maximum system memory
2534  obtained via MORECORE or MMAP.
2535 
2536  Trim support
2537  Fields holding the amount of unused topmost memory that should trigger
2538  timming, and a counter to force periodic scanning to release unused
2539  non-topmost segments.
2540 
2541  Locking
2542  If USE_LOCKS is defined, the "mutex" lock is acquired and released
2543  around every public call using this mspace.
2544 
2545  Extension support
2546  A void* pointer and a size_t field that can be used to help implement
2547  extensions to this malloc.
2548 */

4703 void* dlmalloc(size_t bytes) {
4704  /*
4705  Basic algorithm:
4706  If a small request (< 256 bytes minus per-chunk overhead):
4707  1. If one exists, use a remainderless chunk in associated smallbin.
4708  (Remainderless means that there are too few excess bytes to
4709  represent as a chunk.)
4710  2. If it is big enough, use the dv chunk, which is normally the
4711  chunk adjacent to the one used for the most recent small request.
4712  3. If one exists, split the smallest available chunk in a bin,
4713  saving remainder in dv.
4714  4. If it is big enough, use the top chunk.
4715  5. If available, get memory from system and use it
4716  Otherwise, for a large request:
4717  1. Find the smallest available binned chunk that fits, and use it
4718  if it is better fitting than dv chunk, splitting if necessary.
4719  2. If better fitting than any binned chunk, use the dv chunk.
4720  3. If it is big enough, use the top chunk.
4721  4. If request size >= mmap threshold, try to directly mmap this chunk.
4722  5. If available, get memory from system and use it
4723 
4724  The ugly goto's here ensure that postaction occurs along all paths.
4725  */

4727 #if USE_LOCKS
4728  ensure_initialization(); /* initialize in sys_alloc if not using locks */
4729 #endif

4752  }
4753 
4754  else if (nb > gm->dvsize) {

4757  size_t rsize;
4758  bindex_t i;

4774  replace_dv(gm, r, rsize);
4775  }

4777  check_malloced_chunk(gm, mem, nb);
4778  goto postaction;
4779  }
4780 
4781  else if (gm->treemap != 0 && (mem = tmalloc_small(gm, nb)) != 0) {
4782  check_malloced_chunk(gm, mem, nb);
4783  goto postaction;
4784  }
4785  }
4786  }

4805  }
4806  else { /* exhaust dv */

4809  gm->dv = 0;
4810  set_inuse_and_pinuse(gm, p, dvs);
4811  }
